# 《秘法高塔》（暂名）双 C++ 开发者分工与协作设计

| 字段 | 内容 |
|---|---|
| 团队规模 | 2 人，均为计算机专业学生，主要技能为 C++ |
| 项目性质 | 非商业课程作品 |
| 默认周期 | 8 个迭代；可按课程日期等比例压缩 |
| 成员 A | 已认领（姓名待填写），战斗域负责人 |
| 成员 B | 本局与楼层流程域负责人 |
| 分工原则 | 两人都编写核心 C++、测试和游戏内容，并由对方交叉验收 |

## 1. 为什么采用双系统域分工

两人都擅长 C++，不适合采用“一人写全部程序、另一人只做 UI/素材”的结构。本方案按稳定业务边界拆成两条可并行运行的开发线：

- **成员 A：战斗域。**负责玩家如何移动、攻击、受伤、施法，以及魔法和遗物如何改变战斗。
- **成员 B：流程域。**负责一局如何开始和结束、楼层如何生成和切换、奖励/商店/事件如何推进构筑。
- **共同集成。**Boss 3、技术选型、表现素材、性能优化、试玩和课程答辩由两人共同完成。

这样拆分后，即使另一条工作线尚未完成，A 可以在战斗沙盒中开发，B 可以使用模拟战斗结果开发楼层和奖励流程。

## 2. 课程版范围冻结

为了保证两人能完成完整三 Boss 流程，默认范围冻结如下：

| 内容 | 课程版目标 | 分配 |
|---|---:|---|
| 可操控角色 | 1 | A 主责，B 验收 |
| 普通敌人 | 3 种 | A 负责 1 种，B 负责 2 种 |
| Boss | 3 位 | A 负责 Boss 1，B 负责 Boss 2，Boss 3 共同完成 |
| 普通魔法 | 6 个 | A、B 各实现 3 个 |
| Boss 强力魔法 | 3 个 | Boss 1/2 负责人各实现 1 个，Boss 3 强力魔法共同完成 |
| 遗物 | 6 个 | A、B 各实现 3 个 |
| 商店 | 1 套 | B 主责，A 验收交易规则 |
| 事件 | 2 个 | B 主责流程；A、B 各提供 1 个事件效果 |
| 楼层模板 | 4 类 | B 主责，A 验收战斗层接口 |
| 美术主题 | 1 套主主题 | 两人共同接入占位素材，通过配色变化区分三幕 |

默认不做：复杂局外成长、在线服务、排行榜、多人、Mod、关卡编辑器、多角色、长剧情演出和完整手柄重绑定。

## 3. 成员 A：战斗域负责人

### 长期主责

- Visual Studio 解决方案、项目文件、编译配置和第三方 C++ 依赖接入。
- 平台主循环、输入到 `PlayerIntent` 的转换和战斗时间推进。
- 玩家移动、跳跃、基础攻击、HP、伤害、受击、硬直和死亡。
- 碰撞/命中判定、`DamageResolver` 和战斗结束判定。
- 三魔法槽、施法条件、独立 CD 和魔法效果运行框架。
- 遗物修正阶段、触发器、叠加规则和防递归保护。
- 战斗域单元测试、调试信息和性能分析。
- Boss 1 以及 1 种普通敌人。
- 3 个普通魔法、1 个 Boss 强力魔法和 3 个遗物。

### 向成员 B 提供

- 只读 `PlayerStateView`：HP、金币接口、槽位和冷却状态。
- `CombatRequest/CombatResult`：启动遭遇和报告胜负。
- `CombatEvents`：受击、击杀、玩家死亡和战斗结束事件。
- `SpellDefinition` 与 `RelicDefinition` 的可执行规则和校验器。
- 可独立运行的战斗测试场景。

### 不得单方面修改

- 本局进度、楼层编号、Boss 计数和胜利流程。
- 奖励池、商店、事件和楼层调度规则。
- B 所有的 UI/楼层公共接口。

## 4. 成员 B：本局与楼层流程域负责人

### 长期主责

- `RunController`、`RunState`、顶层游戏流程和第三位 Boss 胜利条件。
- `FloorController`、楼层类型调度、楼层生成、加载、卸载与楼梯事务。
- 确定性 `runSeed/floorSeed`、奖励随机流和固定种子重现。
- 普通/Boss 奖励三选一、已学魔法集合和三槽配置流程。
- 金币、商店交易、事件选择、Boss 进度和跨层回血。
- HUD、奖励、装备、商店、事件、暂停与结果界面的 C++ 状态控制。
- 流程域单元测试、跨层集成测试和可玩冒烟测试。
- Boss 2 以及 2 种普通敌人。
- 3 个普通魔法、1 个 Boss 强力魔法和 3 个遗物。

### 向成员 A 提供

- `RunContext`：当前楼层、幕、Boss 进度和派生种子。
- `FloorDescriptor`：战斗场景需要实例化的纯数据描述。
- `RewardRequest/RewardChoice`：候选生成和奖励应用契约。
- `FloorResult`：当前楼层是否完成、是否可进入楼梯。
- 可跳过战斗或注入假战斗结果的流程测试模式。

### 不得单方面修改

- 伤害、受击、冷却和遗物触发顺序。
- 玩家战斗状态机和命中规则。
- A 所有的战斗公共接口。

## 5. 共同责任

- 决定并验证引擎/2D 框架、测试框架和数据格式。
- 共同设计共享状态结构及接口变更。
- Boss 3：A 负责攻击组件、伤害和施法机制；B 负责阶段流程、场地、奖励和胜利结算。
- 两人共同接入占位美术、动画和音频，不将非程序工作固定压给一人。
- 每人都要在另一台电脑或干净环境中完成一次独立构建。
- 每人都必须理解完整主循环、构建命令和调试入口，确保可以临时接管。
- 共同组织试玩、判定优先级、修复集成问题并完成答辩。

### 工程文件规则

- `.slnx`、`.vcxproj` 和 `.vcxproj.filters` 仍由 A 作为默认合并负责人，目的是防止文本冲突，不代表只有 A 能理解它们。
- B 每个迭代至少运行一次完整 Debug x64 构建；迭代 4 和迭代 7 由 B 按文档从干净环境重建。
- B 新增文件后可提交源文件，由 A 在当天集成时注册到工程；若 A 不可用，B 按构建文档接管。
- 同一场景或不可合并资源同一时间只允许一人编辑，在任务板标记占用者。

## 8. 稳定交接接口

| 契约 | 定义负责人 | 消费方 | 最小内容 |
|---|---|---|---|
| `PlayerIntent` | A | A/B 的输入与 UI | 移动、跳跃、攻击、槽位 0–2、交互 |
| `PlayerStateView` | A | B | HP、金币显示值、槽位、冷却、死亡状态 |
| `CombatRequest` | A+B | A | 敌人列表、场地 ID、难度和种子 |
| `CombatResult` | A+B | B | 胜负、遭遇 ID、金币基础值、击杀信息和剩余 HP |
| `CombatEvents` | A | B | 受击、击杀、玩家死亡、战斗完成 |
| `RunContext` | B | A | 楼层、幕、Boss 进度和种子 |
| `FloorDescriptor` | B | A/表现层 | 场地、出生点、敌人和出口描述 |
| `SpellDefinition` | A | A+B 内容 | 冷却、施法策略、效果列表和表现 ID |
| `RelicDefinition` | A | A+B 内容 | 修正、触发、叠加和表现 ID |
| `RewardRequest/Choice` | B | A/表现层 | 奖励池、候选和选择应用结果 |
| `FloorResult` | B | A | 当前层完成、奖励完成、楼梯可用状态 |

接口确定后一个迭代内不得随意破坏。每个契约必须有字段说明、最小示例、非法输入行为以及一个测试或调试方法。

## 9. 并行开发策略

### A 不等待 B

- 使用固定 `CombatRequest` 在独立战斗沙盒调试玩家、敌人、魔法和遗物。
- 用模拟 `RunContext` 指定楼层难度和 Boss 进度。
- 战斗结束时输出 `CombatResult` 到日志或测试接收器。

### B 不等待 A

- 使用“立即胜利 / 立即失败 / 指定伤害”的假战斗实现推进流程。
- 用固定 `PlayerStateView` 开发 HUD、奖励、商店和装备界面。
- 用脚本化 `CombatResult` 测试金币、奖励、楼梯和 Boss 计数。

### 集成点

- 迭代 1 结束：真实战斗替换假战斗。
- 迭代 2 中期：真实 `PlayerStateView` 接入 HUD 与奖励。
- 迭代 3 结束：双方各自创建的魔法/遗物都能在完整流程中出现。
- 迭代 4 结束：Boss 1 完整纵向切片。
- 迭代 5 结束：Boss 1/2/3 和三幕胜利全部连通。

## 10. 八迭代具体安排

每个迭代默认约一周。如果只有 6 周，可合并迭代 0+1 和迭代 5+6，但不能跳过共同验收。

### 迭代 0：技术选型与双人环境

**成员 A**

- 验证候选框架的窗口、主循环、输入和 C++ 构建接入。
- 初始化 Git、忽略规则、Debug/Release 构建和测试目标。
- 写第一版 `PlayerIntent`、战斗契约和战斗沙盒入口。

**成员 B**

- 验证候选框架的场景切换、2D 碰撞、HUD、资源加载和卸载。
- 写第一版 `RunContext`、`FloorDescriptor`、假战斗和流程测试入口。
- 冻结课程内容量与占位素材目录。

**共同验收**

- 两台电脑能从同一仓库构建并运行。
- 选定技术方案并记录理由。
- 假战斗可以从流程域启动并返回结果。

### 迭代 1：战斗沙盒与最小流程

**成员 A**

- 玩家移动、跳跃、攻击、HP、受击和死亡。
- 伤害解析、魔法槽、一个直接伤害魔法和 CD。
- 普通敌人 1 和战斗规则单元测试。

**成员 B**

- `RunController` 最小状态机、一个固定战斗楼层和楼层进入/退出。
- 基础 HUD、失败状态和流程日志。
- 用假战斗结果验证胜利/失败分支。

**共同验收**

- B 的流程能启动 A 的真实战斗并收到唯一结果。
- 两人互换电脑连续游玩 5 分钟，无阻断和重复结算。

### 迭代 2：单层核心循环

**成员 A**

- 完成三个槽位、已装备状态、冷却快照和第二/第三个魔法效果原语。
- 暴露稳定的 `PlayerStateView` 和 `CombatEvents`。
- 验证帧率变化不改变移动与冷却。

**成员 B**

- 楼层种子、普通奖励三选一、已学魔法和装备界面。
- 楼梯过层事务、50% 已损失 HP 恢复及其测试。
- 第 2 种普通敌人和固定种子下一层。

**共同验收**

- 连续完成 5 次“战斗→奖励→装备→楼梯→回血→下一层”。
- 同一种子在两台电脑产生相同楼层和奖励候选。

### 迭代 3：内容平均分配

**成员 A**

- 完成魔法/遗物运行框架和防递归保护。
- 实现自己的 3 个普通魔法、3 个遗物和 1 个事件效果。
- 为伤害、冷却和遗物触发编写测试。

**成员 B**

- 完成奖励池、金币、商店、事件和楼层类型保底。
- 实现自己的 3 个普通魔法、3 个遗物和 1 个事件效果。
- 完成第 3 种普通敌人及流程测试。

**共同验收**

- 双方各自创建的内容都能通过同一配置/注册方式进入游戏。
- 至少两种不同构筑可完成相同三层流程。
- 商店成功和失败交易都保持金币与库存一致。

### 迭代 4：Boss 1 垂直切片

**成员 A**

- 实现 Boss 1、对应强力魔法和通用 Boss 攻击组件。
- 完成 Boss 阶段、伤害、死亡和性能调试。
- 修复战斗域在完整流程中的集成缺陷。

**成员 B**

- 完成第一幕楼层调度、Boss 场地、Boss 奖励三选一和阶段结算。
- 完成暂停、失败、阶段胜利和必要 UI。
- 组织至少 2 名同学盲玩并记录反馈。

**共同验收**

- 从新局开始能完整击败 Boss 1 并进入下一幕。
- Boss 奖励只发一次，旧场景完全停止更新。
- 根据盲玩结果决定是否继续当前操作和内容范围。

### 迭代 5：Boss 2 与共同 Boss 3

**成员 A**

- 提供 Boss 3 所需攻击组件、伤害机制和最终强力魔法效果。
- 优化 Boss 公共战斗代码，避免复制 Boss 1。
- 验收 Boss 2 对战斗接口的使用和状态清理。

**成员 B**

- 实现 Boss 2、对应强力魔法和第二幕流程。
- 搭建 Boss 3 阶段状态机、最终场地和第三幕胜利结算。
- 验收 Boss 1/3 的流程、奖励与进度。

**共同完成 Boss 3**

- 两人先冻结 Boss 3 的 3–4 个核心招式和阶段条件。
- A 完成招式与战斗逻辑，B 完成阶段编排、场地、表现和结果流程。
- 两人一起调试难度、可读性、资源卸载和第三次 Boss 胜利。

**共同验收**

- 从新局到第三位 Boss 胜利可完整游玩。
- A 验收 Boss 2；B 验收 Boss 1；Boss 3 双人共同验收。

### 迭代 6：稳定性与课程打磨

**成员 A**

- 性能分析、战斗缺陷、崩溃、资源生命周期和 Release 构建。
- 调整自己的敌人、魔法、遗物和 Boss 1。
- 输出战斗域测试和性能数据。

**成员 B**

- 调整流程、商店、奖励、事件、自己的敌人/内容和 Boss 2。
- UI 清晰度、镜头、占位美术/音频和试玩问题整理。
- 完成多种固定种子的全流程回归。

**共同任务**

- 平均分配剩余 P0/P1 Bug，不按原模块拒绝集成问题。
- 每人完整运行至少 3 个种子，其中 1 个必须失败后重新开始。

**共同验收**

- 阻断流程、崩溃、重复奖励、旧层残留和错误胜利全部关闭。
- 两台电脑的 Release 构建均可运行。

### 迭代 7：课程交付

**成员 A**

- 生成游戏包、构建说明、架构图和战斗系统技术说明。
- 准备战斗、确定性规则、测试和性能部分的答辩内容。
- 维护稳定演示种子和备用构建。

**成员 B**

- 整理玩法流程、楼层生成、奖励经济和状态机说明。
- 汇总试玩记录、课程素材来源和操作说明。
- 准备流程域的答辩内容和备用演示路线。

**共同完成**

- PPT 页数和讲解时间均分；两人都必须实际讲解自己的 C++ 模块。
- 共同录制演示视频，每人负责剪辑/讲解其中一半。
- 在第三台电脑完成两次完整彩排。

**共同验收**

- 第三台电脑无需 Visual Studio 即可启动课程游戏包。
- 两人都能独立完成一次从构建、运行到演示种子的操作。
- 现场演示、备用视频和答辩文件均有至少两份独立备份。

## 11. Git 与合并流程

- `main` 始终保持可构建并能进入游戏。
- A 使用 `a/<短任务名>`，B 使用 `b/<短任务名>`。
- 一个分支只解决一个 1–2 天可验收任务。
- 提交前同步 `main`，运行自己的测试并手动走一次相关路径。
- 另一位完成代码检查和实际运行后合并。
- 每天至少合并一次小成果，禁止各自隔离开发一周后大合并。
- 公共契约变更必须单独提交，不与大量功能实现混在一起。

提交示例：

```text
feat(combat): add spell cooldown state
feat(run): add deterministic floor transition
feat(boss1): add second attack phase
feat(boss2): add arena hazard cycle
test(reward): prevent duplicate boss choice
```

## 12. 每日协作节奏

### 开始前 5 分钟

- 各自说清当天交付物、会修改的目录和需要的接口。
- 检查是否会同时编辑工程文件、公共契约或同一场景。
- 将任务放入“待办 / 进行中 / 待验收 / 完成”。

### 结束前 15 分钟

- 合并一个可构建的小成果。
- 在另一位电脑运行相关路径。
- 记录阻塞、接口变化、固定测试种子和下一步。

### 迭代结束 30–60 分钟

- 从干净构建开始演示迭代目标。
- 非主负责人按验收项实际操作。
- 根据剩余周期调整范围，不把未验证功能滚雪球式带入下一阶段。

## 13. 接口变更规则

修改公共契约前先说明：

1. 当前接口为什么不能满足任务。
2. 要增加、删除或修改的字段/事件。
3. 对另一系统域的影响。
4. 迁移步骤和验证方法。

另一位确认后再改。紧急修复可以口头确认，但当天必须更新接口注释或架构文档。

## 14. 交叉验收与完成标准

任务完成必须满足：

- 作者本地构建、测试并运行成功。
- 对方在另一台电脑或干净工作区重现。
- 正常路径和至少一个失败/边界路径已验证。
- 没有绕过公共接口直接修改另一系统域的内部状态。
- 新规则同步到 GDD，新边界同步到架构文档。
- 记录测试命令、测试名、固定种子或演示步骤。

典型交叉验收：

- A 写魔法框架，B 必须能不修改运行时就添加自己的 3 个魔法。
- B 写奖励系统，A 必须验证重复点击不会重复获得魔法或金币。
- A 写 Boss 1，B 必须将其接入第一幕并正确发放奖励。
- B 写 Boss 2，A 必须验证攻击使用统一伤害管线且死亡后不再更新。
- Boss 3 必须由两人各自实现一部分，并共同走查完整代码路径。

## 15. 延期时的砍需求顺序

保留“逐层战斗、三槽魔法、奖励、楼梯、三位 Boss、最终胜利”，按以下顺序缩减：

1. 取消中途存档、手柄重绑定和复杂设置。
2. 三幕共用一套美术主题，通过配色和背景变化区分。
3. 事件固定为 2 个，商店只保留一套界面和商品池。
4. 普通魔法固定 6 个、遗物固定 6 个，不继续追加。
5. 随机关卡降为固定模板 + 随机敌人/奖励。
6. Boss 2/3 复用攻击组件、特效和场地元素，但保留不同阶段机制。
7. 删除剧情演出、复杂局外成长和非必要动画。

不得删除：HP/金币、三个魔法槽、独立 CD、普通奖励三选一、跨层回血、商店、至少一个事件、三次 Boss 和第三次胜利条件。

## 16. 工作量再平衡

- 每个迭代比较剩余工时、阻塞时间和缺陷严重度，不只比较提交数量。
- A 被战斗底层问题阻塞时，B 帮助编写固定重现流程、调试 UI 和集成测试。
- B 的流程/UI任务堆积时，A 接手一个独立流程页面、事件效果或楼层模板。
- A 额外承担一个敌人时，B 应对应接手一个魔法/遗物或战斗测试。
- 任一成员连续两天被对方接口阻塞，立即制作假实现或缩小接口，不等待大系统完成。
- 集成问题由双方共同解决，不能以“不是我的模块”为由长期挂起。

## 17. 课程交付物分工

| 交付物 | 成员 A | 成员 B |
|---|---|---|
| 可执行包 | 主构建与打包 | 干净电脑验收与备用包 |
| 源码说明 | 战斗域、构建与测试 | 流程域、楼层与内容 |
| 架构图 | 战斗依赖与公共契约 | 流程状态机与数据流 |
| 演示视频 | 战斗/Boss 1/技术部分 | 楼层/构筑/Boss 2/流程部分 |
| PPT | 技术选型、战斗和测试页 | 设计、流程和试玩页 |
| 现场讲解 | 战斗域与工程实现 | 流程域与系统集成 |
| Boss 3 展示 | 攻击与伤害机制 | 阶段、场地与胜利流程 |

两人的讲解时间、代码展示量和演示视频时长应大致相当。
