# 《秘法高塔》魔法动画与战斗特效设计

## 1. 文档定位

本文定义当前 24 本普通魔法、9 本 Boss 魔法和固有冲刺的像素动画表现。它只规定视觉语言、动画阶段、资源规格和表现层数据需求，不修改 `SpellSystem`、伤害、冷却、控制时间或权威 AABB。

设计目标：

- 玩家在 0.2 秒内能判断魔法的方向、危险区域和是否命中。
- 同系魔法共享视觉语法，但不同魔法仍可仅凭轮廓和颜色区分。
- 普通魔法保持短促，不遮住敌人前摇；Boss 魔法允许更强演出，但不能覆盖关键攻击提示。
- 用少量可复用精灵表覆盖当前内容池，避免为 33 本魔法分别维护完整粒子系统。
- 所有图片使用最近邻采样、整数像素位置和稳定锚点，符合现有玩家像素风格。

## 2. 画面与资源规范

### 2.1 三层表现

每次施法由下列层按需组合：

1. **判定提示层**：用 SFML 图元绘制低透明度范围、落点或引导线，严格读取领域 AABB。
2. **核心精灵层**：法阵、弹体、光束、斩击、爆炸、召唤物等像素动画。
3. **余波层**：少量火星、冰屑、花瓣、碎石或魔力微粒，只增强反馈，不表达判定。

提示层必须位于角色和敌人下方；核心精灵通常位于实体上方；强闪光和屏幕色调位于最上方。任何精灵透明区域都不能被用于反推命中范围。

### 2.2 标准单元格

| 资源类别 | 单帧尺寸 | 默认锚点 | 用途 |
|---|---:|---|---|
| 手部法阵/施法闪光 | `64×64` | 中心 `(32,32)` | 角色法杖端或手部 |
| 小型弹体 | `64×32` | 左中 `(8,16)` | 丝线头、追踪弹 |
| 中型弹体 | `96×48` | 左中 `(8,24)` | 冰枪、石弹、火焰 |
| 光束端点/枪口 | `64×64` | 中心 | 佐尔特拉克、裁决光 |
| 普通命中特效 | `96×96` | 中心 | 单体命中、近战爆发 |
| 大型命中/终极落雷 | `192×192` | 底中 `(96,176)` | Boss 魔法落点 |
| 自身增益 | `128×128` | 底中 `(64,112)` | 加护、护盾、飞行 |
| 区域循环瓦片 | `64×64` | 中心 | 花田、火焰、重力区域 |
| 召唤物 | `128×96` | 脚底 `(64,92)` | 幻影、魔像、镜像 |

精灵表采用单行从左到右播放；帧间不留可见底色，切帧时每个单元格至少保留 2 px 透明安全边。方向性资源只制作向右版本，向左由 SFML 围绕锚点镜像。

### 2.3 帧率与时长

- 普通启动/命中：6～8 帧，12～16 FPS，总长约 `0.35～0.55 s`。
- 快速弹体循环：4 帧，16～20 FPS。
- 持续区域循环：6 帧，8～10 FPS，循环期间避免大幅明暗闪烁。
- Boss 启动/命中：10～16 帧，12～18 FPS。
- 伤害发生时刻必须由领域事件驱动；美术动画可以提前预警或延后消散，但不能改变结算帧。

### 2.4 混合与亮度

- 轮廓、实体碎片和召唤物使用普通 Alpha 混合。
- 光束中心、雷电、法阵纹路和星点允许加法混合，但外围仍保留实色像素轮廓。
- 普通魔法全屏闪白不超过 `0.04 s / 15%`；Boss 魔法不超过 `0.08 s / 30%`。
- 同屏高亮面积超过画面 25% 时自动降低余波层透明度，保证敌人橙色前摇和红色攻击范围可读。

## 7. 首批资产生产顺序

当前生产状态（2026-07-13）：前七批 32 套、共 266 帧资源已经生成并通过标准化。新增花田及点燃转换、驭血魔法、斩断一切、业火风暴和模仿魔法后，当前 24 个普通魔法与 9 个 Boss 魔法均至少拥有一套核心视觉资源。第一阶段 C++ 接入已经完成：`SpellEffectAnimator` 加载标准化图集，按 `SpellEffectView` 生命周期选帧、按权威范围对齐，并在资源缺失时保留范围框回退。佐尔特拉克与裁决之光的光束主体仍计划由运行时拉伸纹理或程序绘制完成；下一步是事件阶段映射、施法瞬间朝向快照和实机尺寸校准，而不是继续扩充静态素材数量。

第一批不按魔法 ID 顺序制作，而按复用收益排序：

1. `cast-arcane`：通用青白施法法阵，6 帧。
2. `beam-zoltraak-muzzle/body/impact`：覆盖 1004、1017、2001、2010。
3. `projectile-elemental`：冰枪、石弹、追踪弹三个同规格弹体与命中。
4. `burst-forward`：火焰、魔力打击、风压三个轮廓版本。
5. `ring-utility`：扫描、解除、空间破碎、重力预警。
6. `aura-self`：加护、飞行、雷杖、护盾的启动和低亮循环。
7. `tether-seal`：丝线、黄金束缚、封印的弹头与闭环。
8. `target-drop`：浮空坠击、三枪、破灭之雷、大地支配的共享落点。
9. `summon-common`：幻影、石之魔像、镜像阵。
10. 最后制作血刃、斩断一切、业火风暴等不可大量复用的独特资源。

完成前 8 组后，已有 21 本魔法能从彩色矩形占位升级为正式像素特效；剩余资源主要是召唤物和 Boss 独特演出。

## 8. AI 生成提示词基线

每次只生成一种动画，不要求模型同时完成多种效果。推荐使用以下固定前缀：

```text
Create a production-ready 2D pixel-art VFX sprite sheet for a dark fantasy action roguelike.
Side-view gameplay, crisp hard-edged pixels, no anti-aliasing, no blur, no gradients made from
smooth vector shading, limited 16-bit palette, strong readable silhouette. The sheet contains
exactly [FRAME_COUNT] clearly different chronological frames in one horizontal row. Every frame
uses an identical [CELL_WIDTH] x [CELL_HEIGHT] cell, identical anchor and scale. Show real motion:
anticipation, expansion or travel, peak impact, then dissipation. VFX only: no character, enemy,
UI, text, numbers, icons, borders or perspective floor. Pure bright green #00FF00 background,
with no green inside the effect. Keep at least 4 pixels of green padding around every frame.
```

## 9. 表现层数据契约建议

当前 `SpellEffectView { spellId, bounds, remaining, duration }` 足以绘制持续区域，但不足以可靠区分“施法启动、弹体飞行、命中、打断、护盾破裂和特殊联动”。实现正式动画前建议新增只读视觉事件：

```cpp
enum class SpellVisualPhase { Cast, Travel, Telegraph, Impact, Loop, Break, End };

struct SpellVisualEvent
{
    std::uint32_t spellId;
    std::uint64_t sequence;
    SpellVisualPhase phase;
    Vec2 origin;
    Aabb bounds;
    float facingDirection;
    float remaining;
    float duration;
    std::uint32_t variantFlags;
};
```

`variantFlags` 只描述已经由领域层确认的结果，例如冻结、墙击、成功反制、黑冲标记、三枪全中、护盾破裂或斩杀强化。表现层不得通过敌人颜色、距离或当前 HP 再计算这些结果。

## 10. 验收清单

- 同一固定种子战斗中，动画不会改变任何 HP、冷却、位移、控制或命中结果。
- 向左镜像后，枪口、弹体锚点和命中位置仍与权威 AABB 对齐。
- 普通魔法高亮不会遮挡敌人前摇，Boss 魔法期间仍能辨认玩家和 Boss 位置。
- 花田点燃、冰冻、墙击、成功解除、冲刺连携和护盾破裂均有不同于普通命中的视觉反馈。
- 持续效果在低帧率下按 `remaining/duration` 恢复正确阶段，不依赖渲染帧数累计伤害。
- 缺失精灵表时保留当前颜色矩形回退，不得导致施法失败或程序退出。
