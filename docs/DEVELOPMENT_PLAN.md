# 《秘法高塔》（暂名）开发计划

## 1. 执行原则

- 按“可玩的垂直切片”推进，不先搭建完整三幕的空架子。
- 每个里程碑必须有可演示结果和明确验收条件。
- 先验证移动、打击感、三槽普通魔法和逐层循环，再加入 Boss 终极槽并扩大内容量。
- 引擎、测试框架和数据格式属于显式决策，不在普通功能任务中顺手引入。
- 文档中的数量是当前范围基线；增加范围前先评估对内容、美术、测试和平衡的连锁影响。
- 两人具体所有权、迭代交付和交叉验收按 `TEAM_WORK_SPLIT.md` 执行。

## 2. 里程碑总览

| 里程碑 | 可交付结果 | 前置条件 |
|---|---|---|
| M0 技术基线 | 可打开窗口、运行主循环、构建和测试基础设施 | 技术选型批准 |
| M1 战斗沙盒 | 一个房间内移动、攻击、受击并使用一个 CD 魔法 | M0 |
| M2 单层闭环 | 战斗结束、奖励、楼梯回血、卸载并进入新层 | M1 |
| M3 构筑与经济 | 三个普通槽、终极槽、已学魔法、遗物、金币、商店、事件 | M2 |
| M4 第一幕垂直切片 | 多层流程 + Boss 1 + 阶段结算，核心体验可评审 | M3 |
| M5 三幕内容版 | 三位 Boss、完整胜负流程和基础内容池 | M4 通过玩法评审 |
| M6 打磨与发布候选 | 性能、可用性、平衡、存档、打包和回归测试 | M5 |

不提供日历工期估算，直到确定开发人数、引擎、美术生产方式和每周投入时间。

- 第二幕 Boss 雷沃戴的追击速度为 `180 px/s`；轻/重剑气保留 `480/560 px` 有限行程，但前摇改为 `0.32/0.5 秒`，速度均为 `315 px/s`，转向上限约 `66/46 度/秒`，发射前 0.15 秒保持直线，突进收招后还需等待至少 `0.8/1.2 秒`才可再次起手。玩家可用反向走位制造无法立刻修正的转弯半径。二阶段开始时由 5 HP 恢复至完整 300 HP。“四刃交叉处刑”连续释放两轮宽 52 px 的 X 形斩，第一轮锁定起手位置，结算后重新采样玩家位置再预警第二轮；每轮两线共享伤害序列并只造成一次伤害。宽幅剑光由一张横向像素素材旋转复用，交点使用独立 X 爆点。“三剑封途”从三个确定性随机位置显示独立实体剑和 0.55 秒瞄准线，分别射向玩家预测移动中心周围的扇形封锁点，再以 `520 px/s` 沿锁定直线飞行。两轮交叉斩、三飞剑和剑气直线/追踪切换均有战斗测试覆盖。最终 Boss 的菲伦与芙莉莲复制体使用固定 `144 px` 长、`3000 px/s`、最大路径 `1600 px` 的移动光束；它会沿锁定方向飞出屏幕，并以扫掠线段防止高速漏判。菲伦采用 `0.55/0.42 秒`首发/连射前摇，芙莉莲普通光束采用 `0.55 秒`前摇，锁线后均不追踪玩家。
- 第三幕复制体的反应窗口已统一：休塔尔克落地剑气提高至 `32×220 px`，标准跳跃无法跨越而黑冲仍可穿过；菲伦首发/连射前摇改为 `0.7/0.55 秒`、固定锁线且每轮最多三发；芙莉莲落雷起手、地火和普通光束前摇改为 `0.7/0.8/0.6 秒`，保留逐雷落点预警并新增全地面与固定光束预警。对应高度、连射上限和三类预警均由战斗测试覆盖。

## 3. M0：技术基线

### 目标

把空 C++20 控制台项目变成可持续开发的 2D 游戏工程，但不实现大量玩法。

### 工作项

1. 比较并确认引擎或 2D 框架；记录决策、备选方案和放弃理由。
2. 初始化版本控制和忽略规则，排除 `.vs/`、构建输出和本地用户设置。
3. 建立窗口、主循环、日志、输入映射和最小场景。
4. 设定 Debug/Release x64 构建与警告策略。
5. 选择测试框架，添加独立测试目标和一个示例测试。
6. 定义游戏时间、坐标、单位、内容 ID 和错误报告约定。
7. 建立最小资源加载方式和课程素材来源清单。
8. 创建技术探针：角色方块移动、一个碰撞地面、一个可卸载场景。

### 验收

- Debug 与 Release x64 均可从命令行构建。
- 程序能创建窗口、响应退出与基础输入，并干净关闭。
- 测试目标可运行且至少包含一个通过的纯逻辑测试。
- 可加载并卸载一个占位楼层，无悬挂更新或明显资源泄漏。
- 技术选型记录与课程素材来源清单可查。

## 4. M1：战斗沙盒

### 目标

证明基础操作、命中反馈和魔法冷却具有可玩潜力。

### 工作项

- 玩家左右移动、跳跃和基础攻击。
- HP、受击、死亡、击退/硬直的最小规则。
- 训练假人和 1 种可攻击普通敌人。
- 一个直接伤害魔法，绑定到槽位 1，具有独立 CD。
- HUD 显示 HP 与三个槽位；未实现槽位显示为空。
- 统一伤害解析和冷却系统。
- 调试叠层显示碰撞体、速度、状态和帧时间。

### 验收

- 玩家可连续完成移动、跳跃、基础攻击和施法。
- 攻击命中、受击和死亡反馈可辨认，不发生一次命中多次结算。
- 冷却期间无法重复施法，UI 与权威状态一致。
- 帧率变化不会明显改变移动速度或冷却时长。
- 伤害与冷却核心规则具备单元测试。

## 5. M2：单层闭环

### 目标

完成“进入战斗 → 获胜 → 三选一加入已学列表 → 返回地图 → 靠近楼梯交互 → 回血 → 下一层”的核心循环；装备栏作为任意可玩阶段可打开的独立覆盖界面验收。

### 工作项

- `RunController`、`FloorController` 和顶层状态机。
- 第一个普通战楼层模板和第 2 种普通敌人。
- 战斗结束检测与金币发放。
- 至少 4 个功能明显不同的魔法。
- 普通魔法三选一和已学魔法集合。
- 三个普通槽装备界面与合法性校验；终极槽在 Boss 1 里程碑接入。
- 楼梯交互和不可重复提交的过层事务。
- 恢复 50% 已损失 HP，并统一整数取整。
- 使用固定种子生成下一层描述并卸载旧层。

### 验收

- 使用同一种子可重现楼层与奖励候选。
- 三选一只发放一次，三个候选不重复。
- 玩家可把已学魔法装备到任意合法槽位。
- 过层后 HP 恢复公式正确，旧层实体不再更新。
- 可连续通过至少 5 层而不累积明显状态错误。
- 核心循环集成测试覆盖成功路径与重复交互保护。

## 6. M3：构筑与经济

### 目标

让每局出现有意义的构筑差异和资源决策。

### 工作项

- 至少 3 个遗物，其中 1 个风险收益型、1 个魔法联动型。
- 遗物叠加、触发顺序和防递归机制。
- 商店楼层、商品生成、金币交易和购买反馈。
- 1 个事件原型，支持有代价的二选一。
- 普通/商店/事件楼层的权重与保底调度。
- 奖励池耗尽与重复魔法的临时处理方案。
- 构筑信息界面：已学魔法、已装备槽位、遗物和效果摘要。

### 验收

- 商店交易是原子的：金币与商品状态不会只更新一半。
- 风险收益遗物在伤害公式中按文档顺序生效。
- 至少两种明显不同的构筑能通过同一战斗测试房。
- 固定种子下商店、事件与奖励结果可重现。
- 长时间不出现商店/事件的情况受保底规则限制。

## 7. M4：第一幕垂直切片

### 目标

形成可对外试玩和评审的短流程，决定项目是否值得扩充到三幕。

### 内容基线

- 1 套原创环境主题与基础音频。
- 2–4 种普通敌人和至少 2 种组合遭遇。
- 4 个以上普通魔法、至少 3 个遗物。
- 普通战、商店、事件和 Boss 楼层。
- 1 位拥有多阶段或明确机制变化的 Boss。
- Boss 强力魔法三选一。
- Boss 1 奖励后启用终极槽，Boss 魔法只能装备到该槽；任意终极施法共享 18 秒公共冷却。
- 阶段胜利、失败、暂停和设置界面。

### 验收

- 新玩家无需开发者解释即可完成基础移动、施法、奖励和过层。
- 从新局到 Boss 1 胜利或失败可完整游玩，没有阻断流程的问题。
- 至少一次完整流程的崩溃、错误日志和性能数据被记录分析。
- 目标硬件上战斗时达到当前性能预算。
- 试玩反馈能回答：战斗是否清晰、普通三槽是否有决策、终极技能频率是否抢占普通战斗循环、奖励是否值得选择、单层时长是否合适。
- 通过范围评审后才进入 M5。

## 8. M5：三幕内容版

### 目标

将验证过的核心循环扩充为完整一局。

### 工作项

- 以每幕 5 层、全局 15 层为当前基线，通过试玩确定目标局长和难度曲线。
- 增加第二、第三幕环境主题和敌人组合。
- 完成 Boss 2、Boss 3 和第三次 Boss 胜利条件。
- 扩充魔法、强力魔法、遗物、事件与商店池。
- 增加跨幕内容解锁和保底规则。
- 实现胜利/失败统计；是否加入局外成长按已批准设计执行。
- 决定并实现中途存档或明确不支持。

### 验收

- 第三位 Boss 只在正确进度出现，击败后进入胜利结算。
- 一局所有楼层均按需加载，内存不会随楼层数量持续异常增长。
- 内容池不会频繁生成无价值或无法选择的奖励。
- 至少完成多种种子和多种构筑的全流程回归。
- 所有 Boss 奖励、过层回血和失败分支具备自动或可重复验证。

## 9. M6：打磨与发布候选

### 工作项

- 输入缓冲、音效、特效细节，以及已实现镜头震动和命中停顿的试玩调参。
- 键鼠完整可用；若批准手柄支持，则完成重绑定和提示切换。
- 性能分析、加载时间、内存与资源生命周期优化。
- 难度、金币、商店价格、魔法和遗物平衡。
- 辅助功能：音量、屏幕震动开关、视觉提示、文本可读性等。
- 存档迁移/损坏处理、崩溃日志和发布打包。
- 整理课程报告需要的素材来源与参考说明。
- 全流程回归、长时间运行和异常退出测试。

### 验收

- 没有已知阻断流程、存档损坏或稳定复现的崩溃问题。
- 发布构建达到已批准的性能、加载和内存预算。
- 课程展示所用外部素材有可追溯的来源记录，组员电脑均可正常构建和运行。
- 从全新安装到完成/失败一局的路径可重复通过。

## 10. 系统优先级

### P0：必须完成

- 玩家移动、攻击、HP、受击和死亡。
- 三个普通魔法槽、已学普通魔法、独立 CD，以及 Boss 1 后启用的终极槽与公共 CD。
- 普通战与金币/魔法三选一。
- 遗物基础框架。
- 商店与事件至少各一个可玩实现。
- 楼梯、50% 已损失 HP 恢复、按需楼层加载。
- 三次 Boss 与第三次胜利。
- 确定性种子、日志和最小测试体系。

### P1：核心质量

- 多种敌人组合、强力魔法池、遗物联动。
- 保底与奖励池耗尽处理。
- 完整 HUD、配置、商店、暂停和结果界面。
- 数据驱动内容、内容校验、性能调试叠层。
- 设置保存和可访问性选项。

### P2：确认核心体验后再考虑

- 复杂局外成长。
- 每日种子、排行榜或在线服务。
- 大量角色职业、多角色切换。
- 合作/多人游戏。
- 自制关卡编辑器和 Mod 支持。
- 大规模 ECS 或多线程玩法模拟。

## 11. 风险清单

| 风险 | 早期信号 | 缓解措施 |
|---|---|---|
| 未定引擎导致返工 | 领域代码直接依赖临时渲染 API | M0 技术探针；用适配边界隔离 |
| 打击感不足 | 系统完整但试玩无趣 | M1 先评审操作与反馈，不急着扩内容 |
| 随机内容不可解或重复 | 某些种子无法到出口、奖励无价值 | 生成后验证、保底、固定种子回归 |
| 构筑只有数值差异 | 玩家总选同一个魔法 | 设计功能性魔法与遗物联动，记录选择率 |
| 内容规模失控 | 新幕只靠堆敌人/技能数量 | M4 评审通过后才扩三幕；复用组合机制 |
| 遗物触发递归 | 一次命中产生无限效果链 | 明确事件阶段、防重入和链深限制 |
| 过层状态泄漏 | 旧敌人仍更新、内存持续上升 | 明确楼层所有权和卸载集成测试 |
| 素材在另一位组员电脑缺失 | 使用了本地绝对路径或未共享文件 | 统一相对路径、提交占位素材、维护来源清单 |

## 12. 质量门槛

每个里程碑完成前检查：

- Debug x64 构建成功且无新增警告。
- Release x64 在涉及性能或发布路径时成功。
- 相关纯规则具有单元测试，核心流程具有集成或可重复冒烟测试。
- 新内容可通过 ID 查找并通过内容校验。
- 固定种子能重现本次测试场景。
- 没有把用户本地文件、构建产物、未知许可资产加入交付。
- 游戏规则变化已同步到 `GAME_DESIGN.md`，系统边界变化已同步到 `ARCHITECTURE.md`。
- UI 状态流转通过无窗口的 ViewModel 快照测试验证，避免自动化环境依赖图形桌面。
- GitHub Actions 在每次 push 与 pull request 时使用 Windows/Visual Studio 2022 完成 Debug 构建和完整 CTest；CI 通过后方可视为分支可集成。
- 提供独立的 `ui_window_smoke_test` 可视化冒烟测试供现场验收；该目标打开真实 SFML 窗口并自动演示关键 UI 流转，但不加入默认 CTest/CI。
