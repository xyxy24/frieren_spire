# ADR-0001：使用 SFML 3、C++20 与 CMake

- 状态：已接受
- 日期：2026-07-10
- 决策人：课程项目两人小组

## 背景

项目是《C++ 工程与项目设计》课程中的 2D 动作 Roguelike。两位成员均熟悉 C++，需要在有限周期内完成窗口、输入、2D 绘制、音频和资源加载，同时保留足够的自研空间展示 C++ 模块化、RAII、状态机、测试和协作设计。

## 决策

- 使用 C++20 编写游戏和纯逻辑测试。
- 使用 SFML 3.1.0 提供窗口、输入和 2D 图形；音频模块在图形技术探针通过后启用。
- 使用 CMake 作为权威构建系统，并固定 SFML Git 标签以保证可重现。
- 使用 Visual Studio 2026 x64 作为当前 Windows 开发环境。
- SFML 类型只允许出现在平台与表现边界。战斗、流程和内容规则使用项目自有的纯 C++ 类型。

## 理由

- SFML 提供面向 C++ 的模块化接口，适合展示课程要求的工程设计。
- 相比直接使用图形 API，SFML 能减少窗口、输入和渲染样板工作。
- 相比完整引擎，项目仍需自行实现战斗、楼层、奖励、数据与状态机。
- CMake 让两位成员共享同一构建描述，并为测试和后续依赖提供统一入口。

## 后果

- 项目需要维护 SFML/CMake 接入并处理版本更新。
- 碰撞、场景流程、UI 状态和游戏架构仍由项目实现。
- 当前仓库原有的空 Visual Studio `.slnx/.vcxproj` 暂时保留，但不再是权威构建入口；待两位成员确认 CMake 工作流稳定后再决定是否移除。
- 若 SFML 技术探针在一个工作日内不能稳定构建或运行，回退候选为 raylib，并新增 ADR 说明替换原因。
