# Arcane Spire 地图与三幕场景设计

## 1. 设计目标

地图不再全部使用单一平地，但本作的核心仍是动作战斗与魔法构筑，不转型为高难度精密跳台游戏。地图高低差主要服务以下目标：

- 让跳跃、空中施法、冲刺、范围魔法和不同射程产生明确价值。
- 让近战敌人、远程敌人和飞行敌人的组合拥有不同站位压力。
- 用场景结构和色彩区分三幕，而不只替换背景图。
- 保证掉落物、商人、事件 NPC 和楼梯始终可达。
- 所有布局继续由楼层种子稳定复现，并且一次只加载当前层。

首版只实现轴对齐的基础地面、静态单向平台和场景边界。斜坡、攀墙、移动平台、可破坏地形与致死深坑均不属于第一轮地图交付范围。

## 2. 当前尺度与布局约束

当前逻辑画布为 `1280×720 px`，基础地面顶边为 `y=640`。玩家碰撞体为 `42×64 px`，移动速度为 `260 px/s`，理论跳高约 `132 px`，一次完整同高跳跃的理论水平位移约 `211 px`，黑冲位移为 `150 px`。

正式布局采用比理论极限更保守的尺寸：

| 项目 | 首版约束 |
|---|---:|
| 相邻可达平台顶面高度差 | 常规不超过 `96 px`，教学房不超过 `72 px` |
| 必须跳跃跨越的水平空隙 | 常规不超过 `144 px` |
| 单个平台最小宽度 | `160 px` |
| 近战交战平台建议宽度 | 至少 `280 px` |
| 玩家出生安全区 | 至少 `180×96 px`，不得生成敌人或危险物 |
| 楼梯前交互净空 | 至少 `120×96 px` |
| 平台视觉厚度 | `24～32 px`，碰撞只使用顶面 |
| 最高常规平台顶面 | 不高于 `y=352`，避免 HUD 和特效遮挡 |

任意正式布局必须满足：玩家出生点可到达所有强制战斗区域、掉落物可能出现的敌人站位以及楼梯；楼梯不能放在需要特定魔法、飞行或黑冲才能到达的位置。

## 3. 地图基础构件

### 3.1 首版构件

- **基础地面：**贯穿房间底部，保证角色不会因地图机制直接坠出世界。
- **单向平台：**玩家和支持平台导航的地面敌人可站立，从下方向上跳跃时可穿过；玩家站在平台上可按住 `S/下方向` 并按 `Space` 主动向下穿台，平台两侧也必须能走落。
- **装饰前景与背景：**不参与碰撞，不能遮挡玩家、敌人前摇、魔法卡面掉落物和楼梯。
- **出生标记：**区分玩家、地面敌人、高台敌人、飞行敌人、NPC、掉落物回退点和楼梯。
- **战斗边界：**继续使用左右边界限制，不使用看不见的内部空气墙。

### 3.2 暂缓构件

- 冰面惯性、风区、落石、丝线陷阱等环境机制先作为视觉语言，不在第一轮加入伤害或控制规则。
- 致死深坑必须等“坠落伤害 + 安全点复位 + 掉落物回收”三项规则同时具备后才能启用。
- 移动平台必须等玩家、敌人、弹体与特效都能继承平台位移后才能启用。

## 4. 通用竞技场原型

每一幕从同一组可验证的结构原型派生主题版本，而不是完全无约束地随机拼块。

| 原型 | 结构 | 主要用途 |
|---|---|---|
| `SafeRoom` | 完整连续平地，不含任何可站立平台 | 所有商店层与事件层 |
| `FlatIntro` | 宽阔主地面 + 一块低平台 | 教学与需要开阔读招的战斗 |
| `TwinLedges` | 左右各一块低平台，中部主战区 | 远近敌人组合、追踪魔法和换边冲刺 |
| `CentralDais` | 中央宽平台 + 两侧地面 | 范围魔法、包围战和 Boss 召唤物分流 |
| `StaggeredSteps` | 两至三块错层平台，最高差不超过 `96 px` | 跳跃追击、空中魔法和垂直威胁 |
| `UpperGallery` | 一块长高台覆盖部分地面，但两端开放 | 上下层换位、远程压制与近战绕行 |

普通战可使用 `FlatIntro`、`TwinLedges`、`CentralDais`、`StaggeredSteps` 和 `UpperGallery`；商店与事件统一使用纯平地 `SafeRoom`，不生成任何可站立平台；Boss 房使用专门布局，不参与普通随机池。

## 5. 第一幕：阿乌拉与刽子手

### 5.1 视觉与玩法语言

- 地点：格拉纳特领地外缘、被魔族占据的街道、审判庭与断头台广场。
- 主色：冷灰石墙、暗蓝暮色、酒红旗帜、紫色魔力丝线。
- 场景轮廓：城墙、破损屋顶、拱门、断头台、悬挂旗帜与被切断的绳索。
- 地图职责：教会玩家在一至两层高度之间换位，不加入环境伤害。
- 战斗节奏：大部分面积仍是连续地面，高台是战术选择而不是通关障碍。

### 5.2 五层场景

| 层数 | 场景 | 推荐结构 | 设计重点 |
|---:|---|---|---|
| 1 | 暮色城门 | `FlatIntro` | 一块 `64 px` 高的低台，教学移动、跳跃、攻击和冲刺 |
| 2 | 破碎市街 | `TwinLedges` / `CentralDais` | 第一次要求主动换层，保留宽阔中央交战区 |
| 3 | 被遗弃的魔法工坊 | `SafeRoom` | 商店或事件层；全场纯平地，NPC 位于中央，楼梯位于右侧 |
| 4 | 刽子手长廊 | `UpperGallery` / `StaggeredSteps` | 琉格纳、莉涅、德拉特等组合可占据不同高度；最高差 `96 px` |
| 5 | 断头台广场 | 专用 Boss 房 | 宽阔主地面，左右各一块顶面高出主地面 `104 px` 的侧台；平台美术与地面之间必须保留清晰空隙，中央不得放置永久阻挡物，给阿乌拉召唤不死大军留出空间 |

第一幕平台覆盖率不超过画面宽度的 35%，确保玩家先学会战斗，再学习利用地形。

## 6. 第二幕：北部边境

### 6.1 视觉与玩法语言

- 地点：北境关隘、积雪峡谷、废弃哨站、冰封堡垒。
- 主色：深蓝夜空、灰白积雪、青色冰层、橙黄篝火。
- 场景轮廓：雪坡背景、断桥、木制哨塔、冰柱、风雪和远山。
- 地图职责：提高垂直变化和换位频率，突出远程敌人、飞行敌人与范围魔法。
- 环境压力首版只做表现：风雪方向、飘雪密度和冰面反光不改变角色移动参数。

### 6.2 五层场景

| 层数 | 场景 | 推荐结构 | 设计重点 |
|---:|---|---|---|
| 6 | 北境关口 | `TwinLedges` | 两侧雪台制造夹击，但中央仍是连续安全地面 |
| 7 | 风雪断崖 | `StaggeredSteps` | 三段错层，纵向差依次为 `64/96 px`，飞行敌人拥有清晰活动空间 |
| 8 | 废弃哨站 | `SafeRoom` | 商店或事件层；全场纯平地，背景哨塔不参与碰撞，NPC 位于中央篝火旁 |
| 9 | 冰封堡垒桥 | `UpperGallery` / `CentralDais` | 上层远程压制、下层近战追击；两端都可上下换层 |
| 10 | 北境要塞中庭 | 专用 Boss 房 | 主地面宽阔，中央低台不超过 `48 px`；左右边缘各保留至少 `280 px` 决斗空间 |

第二幕平台覆盖率可提高到 45%，但不得通过窄跳或深坑制造非战斗失败。

## 7. 第三幕：一级魔法使选拔

### 7.1 视觉与玩法语言

- 地点：选拔营地、捕获陨铁鸟的林地、地下迷宫、零落王墓式考场与最终评审厅。
- 主色：月白石材、青绿色结界、金色考试纹章、黑紫迷宫阴影。
- 场景轮廓：魔法结界、考试旗帜、古代壁画、晶体灯、镜像门与考官席。
- 地图职责：组合前两幕的结构，让玩家针对敌人编队和自己的构筑选择站位。
- 不再引入新的基础移动规则；难度来自已学结构的组合，而不是突然加入精密跳台。

### 7.2 五层场景

| 层数 | 场景 | 推荐结构 | 设计重点 |
|---:|---|---|---|
| 11 | 第一场考试林地 | `TwinLedges` / `StaggeredSteps` | 树根和岩台形成自然高差，留出大型飞行单位路径 |
| 12 | 结界回廊 | `CentralDais` / `UpperGallery` | 对称结构便于辨认考试规则和敌方施法区域 |
| 13 | 考生休整营地 | `SafeRoom` | 商店或事件层；全场纯平地，作为终局构筑检查与视觉喘息点 |
| 14 | 地下迷宫深处 | `StaggeredSteps` + `UpperGallery` | 三幕最高密度的组合房，但每条路线都能返回主地面 |
| 15 | 最终评审厅 | 专用 Boss 房 | 对称主地面 + 两块 `64 px` 侧台；保持中央视野和终极魔法展示空间，具体 Boss 出生与阶段机关由敌人模块定义 |

第三幕平台覆盖率上限为 50%。视觉上可以复杂，但权威碰撞面必须使用高对比边缘，避免装饰被误认为可站立地形。

## 8. 数据与系统契约

现有 `FloorDescriptor::arenaId` 保留为布局选择结果，但 `100～103` 的占位编号应替换为按幕分类的稳定目录：

- 第一幕：`110～119`
- 第二幕：`210～219`
- 第三幕：`310～319`
- Boss 专用房：`115`、`215`、`315`

布局内容应由数据目录解析为纯数据，不在渲染代码中按楼层号硬编码：

```cpp
struct PlatformSurface
{
    Aabb bounds;
    bool oneWay {true};
    std::uint32_t laneId {};
};

enum class SpawnKind
{
    Player,
    GroundEnemy,
    ElevatedEnemy,
    FlyingEnemy,
    Npc,
    Staircase,
    LootFallback
};

struct ArenaLayout
{
    ContentId id {};
    std::uint32_t act {};
    WorldBounds worldBounds;
    std::vector<PlatformSurface> platforms;
    std::vector<SpawnPoint> spawnPoints;
};
```

字段名可随实现调整，但必须保留以下边界：

- Model 拥有平台碰撞、站立表面和可达性；View 只读取布局绘制主题素材。
- 地图 RNG 只从 `RandomStream::Layout` 取值；增加装饰随机数不得改变遭遇、奖励、商店或事件结果。
- `FloorType::Merchant` 与 `FloorType::Event` 必须直接选择当幕对应的 `SafeRoom`，不得进入战斗竞技场随机池。
- 战斗模块从地图获取带类型的出生点，不自行使用屏幕坐标猜测位置。
- 地面敌人出生点必须携带 `laneId`。在 B 的敌人导航接入平台前，地面敌人只能生成在与主地面连续的 lane；飞行敌人可使用独立空中出生点。
- 最后敌人死亡位置不可达时，魔法书必须回退到 `LootFallback`；不能只依赖原始死亡坐标。

## 9. 生成与验证

地图不是逐块无限随机拼接。每幕维护少量人工审核过的竞技场布局，楼层根据 `act + FloorType + Layout RNG` 确定性选择一个兼容布局，再从标记点选择敌人与 NPC 位置。

进入楼层前至少验证：

1. 玩家出生点、强制战斗 lane、NPC、掉落回退点和楼梯均存在。
2. 玩家不依赖魔法或黑冲即可从出生点到达楼梯。
3. 相邻可达平台符合高度差与水平间隙上限。
4. 出生 AABB 不与平台、NPC、楼梯或其他出生 AABB 重叠。
5. 地面敌人出生在有效 lane 上，碰撞体不会越过平台边缘。
6. 同一 `runSeed + floorIndex` 始终得到相同 `arenaId`、平台和出生点。

## 10. 实施顺序

### M1：第一幕静态平台垂直切片

- 建立 `ArenaLayout`、平台顶面碰撞和布局目录。
- 先实现第一幕 `110～115`，其中至少包含平地、双侧台、中央台、错层台和阿乌拉专用房。
- 玩家可从下方穿过平台并在下落时落在顶面；黑冲不能穿过实体场景边界。
- 用占位色块先验证玩法；正式主题素材接入后仍保留程序化后备绘制。

### M2：出生点与 B 的敌人接口

- 地图向战斗系统提供地面、高台、飞行与 Boss 出生标记。
- B 的地面敌人先限制在所属 lane 内；需要跨层追击的导航作为后续增强。
- 奖励、NPC 与楼梯使用地图标记，不再依赖固定屏幕坐标。

### M3：第二、三幕布局与主题表现

- 在碰撞和出生契约稳定后补齐 `210～215` 与 `310～315`。
- 加入三幕背景、中景、平台边缘和前景装饰，并保证最近邻像素渲染。

### M4：可选环境机制

仅在三幕完整流程稳定后，从冰面、风区、丝线机关、可破坏结界中选择最多一至两项实现，并为伤害、复位和敌人交互补齐测试。

## 11. 验收标准

- 15 层能明显看出三幕主题变化，且每幕至少出现三种不同结构。
- 第一层仍能无说明完成基础移动教学；第十四层能组合高低差但不要求精密跳跃。
- 玩家、敌人、掉落物、NPC 和楼梯不悬空、不嵌入平台且不存在不可达结算物。
- 魔法有效范围与平台布局匹配：近战始终有交战空间，大范围特效不会因贴近画面底边而被裁切。
- 固定种子回放得到相同布局与出生点，Debug 自动测试覆盖平台落地、穿台、边缘离开和可达性验证。

## 12. 当前实现状态

- 已实现三幕共 18 个稳定布局 ID：每幕 1 个纯平地 `SafeRoom`、4 个普通竞技场和 1 个 Boss 专用房。
- `FloorController` 已使用独立 `Layout` 随机流选择普通竞技场；事件、商店和 Boss 使用固定类型布局，不影响遭遇与经济随机流。
- 玩家已支持静态单向平台：上升时穿过、下落时落在顶面、走出水平范围后恢复下落；站在平台上同时按住下方向并按跳跃可主动穿台，基础地面不受此操作影响；平台没有侧面空气墙。
- `ArenaView` 已接入三套独立像素场景纹理：暮色占领城镇、北境雪山要塞、考试大厅与魔法纹章；每幕同时拥有一张透明战斗平台纹理，按布局矩形缩放绘制，视觉厚度不改变权威碰撞顶面。
- 六张运行时纹理由 `ArenaTextures` 在应用启动时加载一次；资源缺失时自动退回程序化背景和色块平台，避免因展示素材缺失阻断课程演示。
- 商店和事件层已确认不生成平台，主题建筑仅作为无碰撞背景。
- 尚未实现地图出生标记和敌人 lane 导航；在 B 接入对应接口前，所有地面敌人继续生成和移动在基础地面，飞行敌人沿用现有空中逻辑。
